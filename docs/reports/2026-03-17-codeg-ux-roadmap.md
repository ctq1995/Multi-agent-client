# Codeg 竞品对标与 UX 迭代路线图（以“快速上手、立刻好用”为中心）

日期：2026-03-17  
面向版本：`v0.1.x` 快速迭代期  
报告目标：用最小的功能增量，最大化“首次体验”和“日常效率”的体感提升；优先解决新用户不知道怎么开始、开始后不够顺滑、价值不够立刻可见的问题。

---

## 1. 结论（TL;DR）

Codeg 当前的差异化不在“再做一个 IDE”，而在“本地优先的 AgentOps for Coding”：

- 把多代理历史会话与项目工程上下文（文件、diff、git、worktree、终端、技能、MCP）连接起来，让用户对“对话产出”可追溯、可复用、可交付。
- UX 上要赢在“开箱即用”：启动后 60 秒内完成从“打开项目”到“开始一次可见产出的对话（并看到文件变更）”。
- 以 P0 把 onboarding 与信息架构打通：让用户清楚“我在哪、我能做什么、下一步是什么、系统现在在做什么”。

---

## 2. 竞品对标（只保留对 UX 有直接价值的观察）

### 2.1 LLM 可观测/LLMOps（Langfuse / LangSmith / PromptLayer / Helicone）

共识 UX 模式：

- “对话/调用 = 资产”：默认可搜索、可过滤、可对比、可导出；不要求用户先理解内部结构。
- “状态可见”：处理进度、失败原因、重试路径清晰；出现异常不会让用户陷入黑盒。
- “指标即价值”：成本、延迟、token、成功率、版本对比是默认展示，不需要用户自己拼凑。

对 codeg 的启示：你们不一定要做云端 LLMOps，但要把本地会话做成“可用资产”而不是“只能看”。

### 2.2 IDE 内建 Agent（Cursor / Windsurf / Copilot / JetBrains Junie）

共识 UX 模式：

- “一条主路径”：打开项目后，默认就能开始对话，且对话能立刻作用于工程（编辑/补全/运行/提交）。
- “可回退、可对比”：变更可视化，随时撤销；用户敢用才会常用。
- “降低心智负担”：把复杂配置藏到后面，先给“能跑起来”的默认体验。

对 codeg 的启示：在你们的优势区（多代理、历史聚合、工程闭环）复刻这些“敢用”机制，而不是追求和 IDE 一样的编辑深度。

### 2.3 工具调用生态（MCP + 市场/目录）

共识 UX 模式：

- “发现与安装”只是第一步，长期壁垒是“权限、供应链、安全与可审计”做成一等能力。
- 安全体验本质也是 UX：用户能不能理解一个服务器会做什么、为什么要授权、出了问题怎么定位。

对 codeg 的启示：你们已经接了 MCP 市场与写入目标应用，下一步应该把“安全可控”做成默认路径，否则规模化会被信任成本拖垮。

## 3. 当前产品快照（基于仓库现状）

你们已经具备的关键能力（这是 UX “快速上手”的基础）：

- Welcome 页：文件夹历史、打开文件夹、克隆仓库入口。
- Folder 窗口：对话区 + 文件工作区 + diff/aux + 终端（可调整布局）。
- 历史会话解析：Claude Code / Codex CLI / OpenCode / Gemini CLI（本地读取并结构化渲染）。
- ACP 实时会话：支持多适配器连接，包含权限请求与事件流。
- Git/Worktree：内建 git 操作、worktree 流程。
- Skills/MCP：支持本地扫描与市场搜索/安装，并写入目标应用配置。

从“上手体感”角度看，主要风险在于信息架构与关键状态呈现：

- 新用户可能不知道“历史会话”和“项目内会话（DB）”的区别、入口在哪里、该先做什么。
- Agent 可用性与连接失败的解释/修复路径如果分散在设置里，会严重拉长“第一次成功对话”的时间。
- 导入/索引/解析失败如果缺乏可见化，会导致用户误判“没用/没数据/不稳定”。

---

## 4. UX 北极星：三条主用户旅程（Time-to-Wow）

### 旅程 A：第一次打开（< 60 秒看到价值）

目标：用户不用读文档即可完成

1. 打开一个项目（或克隆一个仓库）
2. 选择一个 Agent（或自动选择可用默认）
3. 发出第一条指令并得到回复
4. 看到“对话导致的工程变化”或至少看到“可继续工作的明确下一步”

### 旅程 B：日常工作（会话与工程闭环）

目标：用户能用 codeg 作为“对话产出管理器”

- 对话过程中能随手定位文件变更、diff、运行命令、提交 git
- 需要回滚时有安全网（checkpoint / 撤销 / 对比）
- 对话结束后可以沉淀为：commit 信息、变更摘要、可复用的 skill/规则、可追溯的会话链接

### 旅程 C：扩展能力（MCP/Skills）

目标：用户敢装、敢用、出问题能定位

- 市场搜索、安装、配置参数填写路径清晰
- 权限模型直观且可审计
- 与目标应用（Claude/Codex/OpenCode 等）的写入结果可验证

## 5. 迭代路线图（按优先级）

优先级含义：

- P0：直接影响首次成功体验与核心闭环，必须优先。
- P1：显著提升效率与“敢用”程度，紧随其后。
- P2：增强壁垒或团队化能力，适合稳定后投入。

每条建议给出：用户价值、范围、验收标准（Definition of Done），尽量可落地。

---

### P0（让新用户立刻觉得“我会用了，而且有用”）

#### P0-1：Welcome 页做成“上手向导”，把关键状态前置

用户价值：

- 把“打开项目 -> 选 Agent -> 开始对话”变成可视化步骤，减少探索成本。
- 第一次就能知道：哪些 Agent 可用、为什么不可用、点哪里修复。

范围建议：

- Welcome 页新增 “快速开始”卡片（3 步）：
  - Step 1：打开/克隆项目（已有）
  - Step 2：检测可用 Agent（展示可用/不可用原因，提供 Fix 按钮直达设置或一键修复）
  - Step 3：打开该项目并自动创建/打开一个会话
- 将 ACP 的 preflight/agent 可用性结果在 Welcome 页可见（不要要求用户先去 Settings）。

验收标准：

- 冷启动后，用户在 Welcome 页不打开 Settings，也能在 3 次点击内进入一个可对话的项目窗口。
- 若失败，错误信息包含“缺什么/去哪装/点哪里修复”，且有明确 action。

#### P0-2：Folder 空状态优化：没有会话时给“唯一正确的下一步”

用户价值：

- 减少“我进来了但不知道干什么”的空窗。

范围建议：

- 当 Folder 内没有打开的对话 tab / 没有会话记录时：
  - 显示 “开始新对话”主按钮（默认选可用 Agent）
  - 显示 “导入该项目的历史会话”按钮（对应 import_local_conversations）
  - 显示 “为什么我看不到历史会话？” 的解释（历史会话来源与路径）

验收标准：

- 新用户打开一个全新项目时，界面上只有一个主行动按钮（Start Chat），不会出现多入口竞争。
- 导入成功后，侧边栏会话列表立刻可见且可点击打开。

#### P0-3：统一信息架构命名与入口（减少“概念混乱”）

用户价值：

- 把“历史会话聚合”和“项目内会话（DB/工作流）”区分清楚，新用户不再迷路。

范围建议：

- 在 UI 文案和导航上明确两类对象：
  - “历史收件箱（Inbox / History）”：跨 Agent 的本地会话聚合与搜索
  - “项目工作区会话（Workspace Sessions）”：与 folder、git、状态（in_progress/pending_review 等）绑定的会话
- 在 Folder 与 Welcome 都提供直达 Inbox 的入口。

验收标准：

- 用户在 30 秒内能回答：我现在看的是什么会话，它来自哪里，如何回到另一类会话。

#### P0-4：关键进度可见化（导入、解析、索引、连接）

用户价值：

- “好用”的前提是“可预期”：系统正在做什么、要多久、失败怎么处理。

范围建议：

- 对以下动作提供统一进度/结果面板（可在 StatusBar/Toast 中聚合）：
  - 导入本地会话（import）
  - ACP 连接与会话启动
  - MCP 安装/写入目标应用
- 每个动作都落到可复制的日志或详情（方便用户截图/反馈）。

验收标准：

- 任一动作失败时，用户能在 1 个界面内看到失败原因与可执行修复动作。

---

### P1（让老用户“越用越顺手”，形成粘性）

#### P1-1：会话资产化 1.0：强搜索与快速定位

用户价值：

- 用户会把 codeg 当“对话知识库”使用，而不是一次性聊天。

范围建议：

- Inbox/History 支持：
  - 全文检索（至少 text block）
  - 按 Agent、模型、时间范围、项目路径过滤
  - 结果预览（命中高亮、上下文片段）
- Workspace Sessions 支持：
  - 状态/标签快速筛选
  - 一键跳到“本次会话修改过的文件”

验收标准：

- 用户能在 10 秒内从所有会话里找到“那次提到某个关键字/文件名/命令”的上下文。

#### P1-2：安全网（Checkpoint / 回滚 / 对比）让用户“敢用 Agent”

用户价值：

- 让用户放心把修改交给 Agent，不怕工程被弄乱。

范围建议：

- 会话开始时创建一个轻量 checkpoint（git stash 或临时分支/commit，根据你们现有 git 能力选一种）。
- 在对话 UI 中提供：
  - “查看自会话开始以来的变更”
  - “回滚到会话开始”
  - “生成 commit（含摘要草稿）”

验收标准：

- 用户能在不离开 codeg 的情况下完成：对话 -> 查看 diff -> 回滚/提交 的闭环。

#### P1-3：MCP/Skills 的“可理解配置”和“可审计使用”

用户价值：

- 用户安装扩展后能理解风险与收益，出现异常可定位。

范围建议：

- MCP 安装 UI 增强：
  - 参数表单的默认值、示例、是否敏感（secret）清晰
  - 安装后提供“验证按钮”（能发一个最小请求或验证配置写入成功）
- 会话内工具调用日志增强：
  - 记录 MCP server、工具名、输入/输出摘要、错误原因
  - 提供按工具/按 server 过滤

验收标准：

- 用户能回答：某次结果来自哪个 MCP server、用的哪个工具、入参是什么、为什么失败。

---

### P2（更“企业级”的长期壁垒，不影响早期节奏）

#### P2-1：分享与迁移（本地优先）

用户价值：

- 让会话成果更容易复用与传播（尤其是团队内）。

范围建议：

- 会话导出：Markdown/JSON（含脱敏选项）。
- 技能/规则导入导出：支持项目级与全局级迁移。

验收标准：

- 用户能把一次高质量会话一键导出成可读文档，并能在另一台机器导入复现。

#### P2-2：本地评测与对比（轻量 LLMOps 思路）

用户价值：

- 多 Agent 场景下，用户能理性选择：哪个更快/更省/更稳。

范围建议：

- 对同一任务的多次运行做对比卡片：耗时、token、失败率、修改文件数。
- 在本地保存评测结果，不依赖云端。

验收标准：

- 用户能从对比面板快速决定：某类任务以后默认用哪个 Agent/模式。

## 6. “快速上手”专项：建议的 4 个可量化指标

这些指标不需要做复杂埋点系统，也可以先在本地日志/调试面板里统计：

1. Time-to-First-Chat：启动应用到第一次成功收到 assistant 回复的时间（目标 < 60s）。
2. First-Session Success Rate：首次创建会话并成功连接/导入的成功率（目标 > 85%）。
3. Fix Funnel：用户遇到 agent 不可用时，通过 Fix 动作能修复成功的比例（目标 > 60%）。
4. “我看到了价值”代理指标：首次会话中至少发生一次“文件变更可视化/commit/命令执行/导入成功”的比例（目标 > 50%）。
